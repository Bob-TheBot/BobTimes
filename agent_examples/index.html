What are LLM Agents?

LLM Agents are sophisticated AI systems that use Large Language Models as a "brain" or "coordinator" to solve complex tasks through reasoning, planning, memory, and tool usage. Unlike traditional LLMs that simply generate text, agents can autonomously break down problems, execute multi-step plans, and interact with external systems.

üéØ Key Definition

An agent is a system that uses an LLM to determine the control flow of an application, making decisions about what actions to take next based on the current state and goals.

Sequence of Calls

  1. Entry: NewspaperService.run_news_cycle(field_requests) initializes EditorAgent and delegates orchestration.
    File: backend/app/services/newspaper_service.py
  2. EditorAgent orchestrates phases and calls tools in order:
    collect_topics
    Spawns Researcher via TaskExecutionService to gather TopicList per field.
    assign_topics
    Creates TopicAssignment list (reporter IDs + topics).
    collect_story
    Spawns Reporter to write a StoryDraft (can include editor_remarks for revisions).
    review_story
    Basic checks (word_count, sources, summary) ‚Üí approve/revise.
    publish_story
    Persists story to JSON store and triggers image generation.
  3. TaskExecutionService.execute_researcher_task executes ResearcherAgent with retries, using tools below.
  4. TaskExecutionService.execute_reporter_task executes ReporterAgent to produce StoryDraft and images.
  5. Publish: NewspaperFileStore writes to libs/common/data/newspaper.json; images saved under data/images/generated.

Agents & Their Tools

  • assign_topics ‚Äî converts selected topics into TopicAssignments (spawns reporters later).
  • collect_story ‚Äî requests a StoryDraft from a specific reporter/topic.
  • review_story ‚Äî approves or returns for revision based on simple quality gates.
  • publish_story ‚Äî writes approved stories to the newspaper store and generates images.

  • scrape ‚Äî Playwright/Tavily content fetch with interactive content handling.
  • youtube_search ‚Äî Finds videos and transcripts; saves video metadata and sources.
  • fetch_from_memory ‚Äî pulls validated sources for a specific topic from SharedMemoryStore.

  • fetch_from_memory ‚Äî retrieves research for the assigned topic (exact or fuzzy match).
  • use_llm ‚Äî drafts and refines story text using the configured LLM.

    Agent Core: The Brain

    The Agent Core is the central coordination module that acts as the "brain" of the LLM agent, managing all other components and making high-level decisions.

    ‚Ä¢ Goal Definition: Establishes and maintains the agent's objectives
    ‚Ä¢ Tool Access: Provides interface to available tools and APIs
    ‚Ä¢ Planning Coordination: Explains how to use planning modules
    ‚Ä¢ Memory Integration: Incorporates relevant historical context
    ‚Ä¢ Persona (Optional): Defines agent personality and behavior patterns
    # Agent Core Example class AgentCore: def __init__(self): self.goals = [] self.available_tools = [] self.memory = MemoryModule() self.planner = PlanningModule() self.persona = "helpful assistant" def coordinate(self, user_input): # Central decision making context = self.memory.retrieve_relevant() plan = self.planner.create_plan(user_input) return self.execute_plan(plan, context)

    • ‚Ä¢ External vector stores for retrieval
    • ‚Ä¢ Semantic similarity matching
    • ‚Ä¢ Importance and recency weighting

    Tools: External Capabilities

    Tools are well-defined executable workflows or specialized third-party APIs that extend the agent's capabilities beyond text generation.

    ‚Ä¢ RAG Pipeline: Retrieval-augmented generation for knowledge access
    ‚Ä¢ Code Interpreter: Execute and debug code in various languages

    ‚Ä¢ Messaging APIs: Communication with external systems

    Tool Integration Methods

    MRKL (Modular Reasoning, Knowledge and Language)

    OpenAI's framework combining LLMs with external tools and knowledge bases. Allows models to call APIs, calculators, and databases.

    Toolformer

    Meta's approach to teach LLMs when and how to use external tools through self-supervised learning, without human annotations.

    Function Calling

    Native capability in modern LLMs (GPT-4, Claude) to call predefined functions with structured parameters and handle responses.

    HuggingGPT

    Microsoft's system that uses ChatGPT as a controller to coordinate multiple AI models from Hugging Face for complex multimodal tasks.

    # Tool Integration Example class ToolManager: def __init__(self): self.tools = { "search": SearchAPI(), "weather": WeatherAPI(), "code": CodeInterpreter(), "rag": RAGPipeline() } def execute_tool(self, tool_name, params): if tool_name in self.tools: return self.tools[tool_name].execute(params) else: raise ValueError(f"Tool {tool_name} not found") # Usage in agent result = tool_manager.execute_tool("search", { "query": "latest AI research", "limit": 5 })
    üìã

    Planning Module: Strategic Thinking

    The Planning Module addresses complex problems by breaking them down into simpler, manageable sub-tasks and creating execution strategies.

    Planning Without Feedback

    Static decomposition approaches:

    • ‚Ä¢ Chain of Thought: Sequential reasoning steps
    • ‚Ä¢ Tree of Thoughts: Branching exploration of possibilities
    • ‚Ä¢ Task Decomposition: Breaking complex tasks into subtasks

    Planning With Feedback

    Iterative refinement approaches:

    • ‚Ä¢ ReAct: Reasoning and Acting - Thought ‚Üí Action ‚Üí Observation cycle for systematic problem solving
    • ‚Ä¢ Reflexion: MIT/Princeton research method where agents reflect on failures and learn from past mistakes to improve future performance
    • ‚Ä¢ Critic Mechanisms: Self-evaluation systems that assess and improve agent outputs through iterative feedback

    ReAct Planning Cycle

from langchain import hub
from langchain.agents import AgentExecutor, create_react_agent
from langchain.memory import ConversationBufferMemory
from langchain_core.tools import tool
from langchain_openai import ChatOpenAI

# Define custom tool
@tool
def get_user_age(name: str) -> str:
    """Use this tool to find the user's age."""
    if "bob" in name.lower():
        return "42 years old"
    return "41 years old"

# Initialize model and memory
model = ChatOpenAI(temperature=0)
memory = ConversationBufferMemory(
    memory_key="chat_history",
    return_messages=True
)

# Create agent with tools and memory
tools = [get_user_age]
prompt = hub.pull("hwchase17/react")
agent = create_react_agent(model, tools, prompt)

agent_executor = AgentExecutor(
    agent=agent,
    tools=tools,
    memory=memory,  # Persistent memory
    verbose=True
)

# Execute with memory
result = agent_executor.invoke({
    "input": "Hi! My name is Bob. What is my age?"
})

from langgraph_supervisor import create_supervisor
from langchain_openai import ChatOpenAI

# Define specialized agents
def book_flight(from_airport: str, to_airport: str):
    """Book a flight"""
    return f"Booked flight from {from_airport} to {to_airport}"

def book_hotel(hotel_name: str):
    """Book a hotel"""
    return f"Booked stay at {hotel_name}"

# Create specialized agents
flight_assistant = create_react_agent(
    model="openai:gpt-4o",
    tools=[book_flight],
    prompt="You are a flight booking assistant",
    name="flight_assistant"
)

hotel_assistant = create_react_agent(
    model="openai:gpt-4o",
    tools=[book_hotel],
    prompt="You are a hotel booking assistant",
    name="hotel_assistant"
)

# Create supervisor to coordinate agents
supervisor = create_supervisor(
    agents=[flight_assistant, hotel_assistant],
    model=ChatOpenAI(model="gpt-4o"),
    prompt="Coordinate hotel and flight bookings"
).compile()

# Execute multi-agent workflow
for chunk in supervisor.stream({
    "messages": [{
        "role": "user",
        "content": "Book flight BOS->JFK and McKittrick Hotel"
    }]
}):
    print(chunk)

    Multi-Agent Architecture Patterns

    Modern frameworks like LangGraph enable sophisticated multi-agent architectures with distinct coordination strategies

    üëë

    Supervisor Pattern

    Single-level management: One supervisor agent directly manages and coordinates multiple worker agents in a flat organizational structure

    Key Characteristics:
    • ‚Ä¢ Centralized control: Supervisor makes all strategic decisions
    • ‚Ä¢ Direct supervision: Maintains ongoing relationships with workers
    • ‚Ä¢ Task delegation: Assigns work based on worker capabilities
    • ‚Ä¢ Coordination focus: Manages dependencies between tasks
    vs Orchestrator-Worker:
    • ‚Ä¢ Supervisor maintains stateful relationships with workers
    • ‚Ä¢ Workers may have ongoing conversations with supervisor
    • ‚Ä¢ Focus on coordination and management rather than just execution
    • ‚Ä¢ Supervisor can reassign tasks and adapt plans dynamically
    Use cases: Project management, team coordination, workflow orchestration, newsroom editorial management
    ‚úì Clear hierarchy and accountability
    ‚úì Easy coordination and communication
    ‚úó Single point of failure
    ‚úó Supervisor can become overwhelmed
    ‚ö°

    Orchestrator-Worker

    Parallel task execution: Central orchestrator spawns multiple independent workers to process tasks concurrently, then aggregates results

    Key Characteristics:
    • ‚Ä¢ Parallel processing: Multiple workers execute simultaneously
    • ‚Ä¢ Stateless workers: Workers are independent and don't maintain state
    • ‚Ä¢ Result aggregation: Orchestrator combines outputs from all workers
    • ‚Ä¢ Batch execution: Optimized for high-throughput processing
    vs Supervisor Pattern:
    • ‚Ä¢ Workers are stateless and independent
    • ‚Ä¢ Focus on parallel execution rather than coordination
    • ‚Ä¢ Fire-and-forget task assignment model
    • ‚Ä¢ Orchestrator aggregates results rather than managing relationships
    Use cases: Data processing pipelines, batch operations, map-reduce systems, parallel research tasks
    ‚úì High throughput via parallelization
    ‚úì Scalable worker pool
    ‚úó Orchestrator can become bottleneck
    ‚úó Complex result aggregation logic
    üêù

    Swarm Pattern

    Dynamic expertise routing: Agents autonomously decide which specialist should handle the next task based on capabilities and context

    Key Characteristics:
    • ‚Ä¢ Self-organizing: No central coordinator, agents choose next steps
    • ‚Ä¢ Expertise-based routing: Decisions based on agent capabilities
    • ‚Ä¢ Emergent behavior: Complex solutions emerge from simple interactions
    • ‚Ä¢ Adaptive routing: Agents learn optimal handoff patterns
    vs Pipeline Pattern:
    • ‚Ä¢ Dynamic routing vs predetermined sequence
    • ‚Ä¢ Agents can skip stages or route to any specialist
    • ‚Ä¢ Adaptive behavior based on context and expertise
    • ‚Ä¢ Non-linear flow allows for complex problem-solving paths
    Use cases: Medical diagnosis teams, technical support routing, adaptive problem-solving, research collaboration
    ‚úì Highly adaptive and resilient
    ‚úì Emergent intelligence from collective behavior
    ‚úó Unpredictable execution paths
    ‚úó Difficult to debug and control
    üîÑ

    Pipeline Pattern

    Sequential processing chain: Agents form a linear sequence where each agent processes and passes results to the next in a predetermined order

    Key Characteristics:
    • ‚Ä¢ Linear data flow: Information flows in one direction through stages
    • ‚Ä¢ Stage specialization: Each agent handles specific transformation
    • ‚Ä¢ Predictable order: Execution sequence is predetermined
    • ‚Ä¢ Sequential processing: Each stage completes before the next begins
    vs Swarm Pattern:
    • ‚Ä¢ Fixed sequence vs dynamic routing
    • ‚Ä¢ Predictable flow makes debugging easier
    • ‚Ä¢ Stage specialization vs expertise-based routing
    • ‚Ä¢ Linear progression ensures all steps are completed
    Use cases: Assembly lines, content processing workflows, validation chains, document processing
    ‚úì Clear data flow and easy debugging
    ‚úì Specialized processing at each stage
    ‚úó Sequential bottlenecks limit throughput
    ‚úó Failure in one stage affects entire pipeline

    Interactive Quiz

    Test your understanding of LLM agents with these comprehensive questions

    1. Fundamental Differences

    Question: How do LLM-powered agents fundamentally differ from a basic Retrieval-Augmented Generation (RAG) pipeline?

    Answer: LLM-powered agents go beyond simple information retrieval by incorporating complex reasoning, planning, memory, and tool-use capabilities. Unlike RAG, which primarily augments LLMs with external data for answering direct questions, agents can break down complex problems into sub-tasks and execute plans using various tools.

    2. Agent Core Function

    Question: Describe the primary function of the "agent core" within an LLM agent's architecture.

    Answer: The agent core is the central coordination and decision-making module of an LLM agent. It manages the agent's core logic, defines its goals, provides access to available tools, and explains how to utilize different planning modules.

    3. Memory Systems

    Question: Explain the distinction between short-term and long-term memory in the context of LLM agents.

    Answer: Short-term memory refers to an agent's "train of thought" for a single user question, limited by the LLM's context window. Long-term memory, on the other hand, stores a cumulative log of past interactions and internal thoughts over extended periods, often relying on external vector stores for retrieval.

    4. Tool Examples

    Question: Provide two examples of "tools" that an LLM agent might utilize, and briefly explain their purpose.

    Answer: An LLM agent might use a "code interpreter" to solve programmatic tasks or generate charts from data, or a "search API" to retrieve information from the internet. These tools enable the agent to interact with external environments and perform actions beyond its inherent knowledge.

    5. Planning Module

    Question: What is the main goal of the planning module's "task and question decomposition" technique?

    Answer: The main goal of the planning module's "task and question decomposition" is to break down complex or compound questions into multiple, simpler sub-questions or tasks. This systematic breakdown allows the agent to address each part individually, making the overall problem more manageable and solvable.

    Glossary of Key Terms

    Essential terminology for understanding LLM agents and their components

    LLM Agent

    A system that uses a Large Language Model (LLM) to reason through a problem, create a plan to solve it, and execute the plan with the help of various tools, memory, and planning modules.

    Retrieval-Augmented Generation (RAG)

    A technique that enhances LLMs by retrieving relevant information from an external knowledge base to generate more informed and context-aware answers.

    Agent Core (Agent Brain)

    The central coordination module and decision-making unit of an LLM agent, managing its logic, goals, tools, and planning modules.

    Memory Module

    A component that stores an agent's internal logs, past thoughts, actions, observations, and interactions with users, crucial for maintaining context and learning.

    Tools

    Well-defined executable workflows or specialized third-party APIs that an LLM agent can use to interact with external environments and execute tasks (e.g., code interpreter, search API).

    Planning Module

    A component that enables an LLM agent to break down complex problems into simpler sub-tasks and strategically decide on future actions.

    ReAct

    A popular prompting framework that interleaves reasoning (Thought) and acting (Action) steps, receiving feedback from the environment (Observation) to solve complex tasks.

    Reflexion

    A research framework developed by MIT and Princeton that enables agents to reflect on their past failures and learn from mistakes. The agent maintains a memory of previous attempts and uses this experience to improve future performance through self-reflection and iterative refinement.

    MRKL (Modular Reasoning, Knowledge and Language)

    OpenAI's framework for combining large language models with external tools and knowledge bases. MRKL systems can call APIs, use calculators, access databases, and integrate various specialized modules to solve complex tasks that require both reasoning and external information.

    Toolformer

    Meta's approach to teaching language models when and how to use external tools through self-supervised learning. Unlike other methods, Toolformer learns tool usage without requiring human annotations, automatically discovering which tools to use for different types of problems.

    HuggingGPT

    Microsoft's system that uses ChatGPT as a central controller to coordinate multiple AI models from Hugging Face. It can handle complex multimodal tasks by planning which models to use, executing them in sequence, and integrating their outputs to solve sophisticated problems.

    Swarm of Agents

    A collection of multiple LLM agents working collaboratively in a shared environment to solve complex problems, often used for simulations or multi-faceted tasks.

    Multi-modal Agents

    LLM agents capable of processing and responding to various types of input data beyond text, such as images, audio, and video.

    Prompt Robustness

    The ability of an LLM agent's performance to remain stable and reliable despite slight variations or changes in its input prompts.

    Knowledge Boundary

    The scope and limitations of an LLM's internal knowledge, which can affect an agent's behavior and potentially introduce biases or hallucination if not managed effectively.
